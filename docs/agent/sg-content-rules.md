# SG Content Rules

`pages/sg` 配下で情報セキュリティマネジメント試験（SG試験）向けの記事を作成・修正するときの基本ルールです。

詳細は次の関連ファイルも必ず確認してください。

- 通常記事の構成: `docs/agent/sg-article-template.md`
- front matter: `docs/agent/sg-frontmatter-rules.md`
- タグ: `docs/agent/sg-tag-rules.md`
- 過去問・例題・確認問題: `docs/agent/sg-example-question-rules.md`
- シリーズまとめページ: `docs/agent/sg-series-summary-rules.md`

## 基本姿勢

- 初学者にも分かる日本語で書く。
- 数式は原則使わない。
- 用語の丸暗記ではなく、判断基準が残る説明にする。
- 「なぜその選択肢が誤りか」を言語化する。
- 1ページは「1用語・1概念」に集中する。
- 単体記事として読んでも理解できるようにする。
- あとで index ページやカテゴリページからリンクされる前提で、タイトル、description、本文を作る。
- 既存の `pages/sg` の記事と、構成、粒度、言い回しをそろえる。

## SG試験向けの説明方針

SG記事では、技術や制度の網羅的な解説よりも、試験で選択肢を切る判断基準を優先します。

- 情報セキュリティマネジメント試験（SG試験）向けの学習記事として書く。
- 技術や制度の深掘りよりも、試験で問われる判断軸を明確にする。
- 業務での判断、現場運用、リスク対応、教育、委託先管理の文脈で説明する。
- 法令・規格は細部暗記ではなく「何を守るためのルールか」を中心に説明する。
- 科目Aの知識と科目Bのケース問題を橋渡しする説明にする。

本文中では、必要に応じて次のような表現を自然に入れます。

- 「SG試験では〜と問われることが多い」
- 「選択肢では〜と書かれていたら注意」
- 「〜と〜を混同させてくる」
- 「判断するときは、〜を見る」

ただし、不自然なSEO文や定型文の使い回しは避けます。

## 混同防止

通常記事では、必ず次のいずれかを本文に含めます。

- 似た用語との違い。
- よくある誤解。
- SG試験での典型的なひっかけ。
- 選択肢を切る判断軸。
- 実務では正しそうに見えるが、試験では誤りになりやすいポイント。

特に `## よくある誤解・混同` では、「この表現なら誤り」と判断できる形にします。

## 公式リンク・参考リンク

SG記事では、公式情報がある用語では公式リンクを1つ以上入れます。

対象例:

- IPA
- NISC
- JPCERT/CC
- CRYPTREC
- 個人情報保護委員会
- e-Gov法令検索
- JIS / ISO / IEC 関連の公式情報

リンク設置ルール:

- 公式リンクは `## 定義・仕組み` または `## どんな場面で使う？` に自然に入れる。
- リンク集のようにしない。
- 記事の主役は、あくまでSG試験対策の説明にする。
- 公式情報が確認できない場合は、無理にリンクを作らない。
- 外部リンクは通常のMarkdown形式にする。

例:

```md
公式な出題範囲は、IPAの[情報セキュリティマネジメント試験の試験内容](https://www.ipa.go.jp/shiken/kubun/sg/outline.html)でも確認できます。
```

## 新規SG記事を作成するときの手順

1. ユーザーの依頼から、記事化する用語・概念を特定する。
2. 既存の `pages/sg` に同じ記事、近い記事、まとめ記事がないか確認する。
3. 既存記事がある場合は、新規作成ではなく追記・ブラッシュアップを提案または実行する。
4. 新規作成する場合は、permalink に合わせた英語スラッグを決める。
5. `pages/sg/英語スラッグ.md` を作成する。
6. front matter を標準形式で書く。
7. 本文を6見出し固定で作る。
8. 公式リンクが必要な用語では公式情報を確認して入れる。
9. 最後に `{% include sg_article_footer.html %}` を入れる。
10. Markdownとして崩れていないか確認する。
11. `tags` が3〜5個で、`sg` と主分類タグを含んでいるか確認する。
12. `last_modified_at` が入っているか確認する。
13. 既存記事と粒度・言い回しが浮いていないか確認する。

## 既存SG記事を修正するときの手順

1. 対象記事を読む。
2. 近いテーマの記事も確認する。
3. 既存記事の構成を壊さず、SG標準の6見出しに寄せる。
4. 添付問題やユーザーの誤解がある場合は、`## よくある誤解・混同` を強化する。
5. 公式リンクを追加できる場合は、自然な位置に追加する。
6. front matter の `description`、`tags`、`last_modified_at` を確認する。
7. 必要に応じて `last_modified_at` を更新する。
8. 変更内容を最後に簡潔に説明する。

## 注意点

- 記事本文に、作業メモやCodex向けコメントを入れない。
- 記事本文に、不要な関連記事セクションを追加しない。
- `prev` / `next` を推測で入れない。
- `tags` を文字列にしない。
- front matter の前に空行を入れない。
- 公式リンクが見つからない場合に、存在しないURLを作らない。
- 過去問画像をそのまま引用する記事にしない。
- 既存記事と重複する新規記事を安易に作らない。

## 最終報告

SG記事やSGルールを編集した場合は、最終報告で次を簡潔に伝えます。

- 作成または更新したファイル。
- 反映した主なルール。
- 確認したポイント。
- front matter、tags、description、6見出し構成、フッター、公式リンクの有無。
