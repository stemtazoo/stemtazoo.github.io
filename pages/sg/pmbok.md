## まず結論

- **PMBOK**とは、プロジェクトマネジメントに関する知識や実務上の考え方を整理したガイドです。
- SG試験では、PMBOKの細かい版や用語を暗記するよりも、**プロジェクトで何を管理するのか**を判断できることが重要です。
- 選択肢では、**スコープ、タイム、コスト、品質、リスク、コミュニケーション**などの管理対象を混同させてくることがあります。

PMBOKは、Project Management Body of Knowledge の略です。

日本語では、プロジェクトマネジメント知識体系のように説明されることがあります。

ただし、SG試験対策では、言葉を難しく覚えるよりも、**プロジェクトをうまく進めるための管理の考え方を整理したもの**と押さえると理解しやすいです。

---

## 直感的な説明

PMBOKは、プロジェクト管理のための **道具箱** のようなものです。

プロジェクトでは、いろいろな問題が起こります。

- 作業範囲があいまいになる
- 予定より遅れる
- 予算を超える
- 関係者への連絡が不足する
- 想定していないリスクが起こる
- 完成物の品質が期待と合わない

このような問題に対して、場当たり的に対応するのではなく、

- 何を決めるべきか
- 何を管理すべきか
- どの観点で確認すべきか

を整理してくれる考え方がPMBOKです。

たとえば、旅行の計画で考えると分かりやすいです。

| 旅行で考えること | プロジェクトでの考え方 |
|---|---|
| どこまで行くか | スコープ管理 |
| いつ出発し、いつ帰るか | タイム管理 |
| いくら使うか | コスト管理 |
| 誰と相談するか | ステークホルダ管理 |
| トラブルにどう備えるか | リスク管理 |
| 必要な予約をどう取るか | 調達管理 |

PMBOKは、こうした管理の観点を整理して、プロジェクトを進めやすくするための考え方です。

## 定義・仕組み

PMBOKは、米国のPMIが提供しているプロジェクトマネジメントに関するガイドです。

PMI公式ページでは、PMBOK Guide はプロジェクトマネージャが成果を上げるために必要な基本的な実務をまとめたものとして紹介されています。詳しくは [PMI公式のPMBOK Guideページ](https://www.pmi.org/standards/pmbok) を確認できます。

PMBOKをSG試験向けに理解する場合、中心になるのは **プロジェクトを管理する観点** です。

代表的には、次のような管理対象があります。

| 管理対象 | 見分けるポイント |
|---|---|
| 統合 | 全体をまとめて調整する |
| ステークホルダ | 関係者の期待や影響を管理する |
| スコープ | 作業範囲や成果物の範囲を管理する |
| 資源 | 人員・設備などを管理する |
| タイム | 作業順序やスケジュールを管理する |
| コスト | 予算や費用を管理する |
| リスク | 起こるかもしれない問題に備える |
| 品質 | 求められる水準を満たすようにする |
| 調達 | 外部から必要なものを入手する |
| コミュニケーション | 情報共有の方法を管理する |

ここで大事なのは、表を丸暗記することではありません。

SG試験では、問題文に出てくる状況を読んで、

- 作業範囲の話か
- 日程の話か
- 費用の話か
- リスクの話か
- 関係者の話か
- 情報共有の話か

を見分けることが大切です。

## どんな場面で使う？

PMBOKの考え方は、次のようなプロジェクトを進める場面で使います。

- 新しいシステムを導入する
- 情報セキュリティ対策を全社に展開する
- ネットワーク構成を変更する
- 外部委託先と協力してシステムを構築する
- 社内ルールや運用手順を新しく整備する

SG試験では、PMBOKそのものを専門資格のように深く問うというより、**業務上の管理判断**として出てくることが多いです。

たとえば、新しいセキュリティ教育を導入するプロジェクトでは、次のように考えます。

| 状況 | PMBOK的に見る観点 |
|---|---|
| 教育対象をどこまでにするか | スコープ |
| いつ実施するか | タイム |
| 外部講師費はいくらか | コスト |
| 教育内容が目的に合っているか | 品質 |
| 受講率が低い可能性がある | リスク |
| 部門長へどう周知するか | コミュニケーション |

このように、PMBOKは「管理項目を増やすためのもの」ではなく、**見落としを減らし、プロジェクトを目的に向けて進めるための整理方法**です。

## よくある誤解・混同

### 誤解1：PMBOKは手順書そのものである

PMBOKは、特定の会社や特定の作業だけに使う細かい手順書ではありません。

プロジェクトマネジメントで考えるべき知識や観点を整理したガイドです。

選択肢で「PMBOKはすべてのプロジェクトで同じ手順を強制するもの」といった説明があれば注意です。

### 誤解2：PMBOKを覚えればプロジェクトは必ず成功する

PMBOKは役立つ考え方ですが、使えば必ず成功するというものではありません。

実際のプロジェクトでは、組織の状況、関係者、予算、技術的制約などに合わせて管理する必要があります。

SG試験では、「ガイドであるもの」を「成功を保証するもの」と言い切る選択肢に注意します。

### 誤解3：スコープとタイムを混同する

スコープは、**何をどこまで行うか**を管理します。

タイムは、**いつ、どの順序で、どれくらいの期間で行うか**を管理します。

作業範囲の変更ならスコープ、作業順序や所要日数ならタイム、と切り分けます。

### 誤解4：ステークホルダ管理とコミュニケーション管理を混同する

ステークホルダ管理は、関係者の期待や影響を管理する考え方です。

コミュニケーション管理は、情報を誰に、いつ、どの方法で伝えるかを管理する考え方です。

どちらも人に関係しますが、試験では次のように切り分けます。

- 関係者の利害や期待に注目する → ステークホルダ管理
- 情報共有の方法に注目する → コミュニケーション管理

### 誤解5：リスク管理とインシデント対応を同じものと考える

リスク管理は、まだ起きていない不確実な問題に備える考え方です。

一方、インシデント対応は、すでに起きた問題に対応する活動です。

選択肢で、起きる前の備えなのか、起きた後の対応なのかを確認すると判断しやすくなります。

---

## まとめ（試験直前用）

- PMBOKは、プロジェクトマネジメントの知識や実務上の考え方を整理したガイドです。
- SG試験では、PMBOKの細かい版や専門用語より、**何を管理する話か**を見分けることが大切です。
- 作業範囲ならスコープ、日程ならタイム、費用ならコスト、不確実性ならリスクです。
- 関係者の期待はステークホルダ管理、情報共有の方法はコミュニケーション管理として切り分けます。
- PMBOKは成功を保証するものではなく、プロジェクトを管理するための考え方を整理したものです。
