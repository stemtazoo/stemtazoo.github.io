## まず結論

- **プロジェクトマネジメント**は、限られた時間・費用・人員の中で、目的に合った成果物を完成させるための管理です。
- SG試験では、PMBOKの細かい暗記よりも、**何を管理している話か**を見分けることが重要です。
- このページでは、PMBOKの管理領域、WBS、アローダイアグラム、クリティカルパス、ダミー作業をまとめて整理します。

プロジェクトマネジメント分野では、似た用語が多く出てきます。

たとえば、

- スコープマネジメント
- タイムマネジメント
- コストマネジメント
- リスクマネジメント
- 品質マネジメント
- ステークホルダマネジメント
- コミュニケーションマネジメント

などです。

これらを丸暗記しようとすると混乱しやすいです。

SG試験では、**範囲の話なのか、日程の話なのか、費用の話なのか、関係者の話なのか**を切り分けると、選択肢を判断しやすくなります。

## プロジェクトマネジメントの全体像

プロジェクトとは、日常業務のように繰り返し続く作業ではなく、**一時的に行われ、独自の成果物を生み出す活動**です。

たとえば、次のような活動はプロジェクトとして考えられます。

- 新しいシステムを導入する
- 多要素認証を全社へ展開する
- セキュリティ教育の仕組みを新しく作る
- 委託先管理のルールを整備する
- ネットワーク構成を変更する

プロジェクトでは、目的、作業範囲、日程、費用、品質、リスク、関係者などをバランスよく管理する必要があります。

1つを変更すると、他の要素にも影響します。

たとえば、作業範囲を増やすと、スケジュールが延びたり、コストが増えたり、品質確認の作業が増えたりします。

そのため、プロジェクトマネジメントでは、個別の作業だけでなく、全体を見ながら調整することが重要です。

## PMBOKの管理領域

PMBOKは、プロジェクトマネジメントに関する知識や実務上の考え方を整理したガイドです。

PMBOK Guide については、[PMI公式のPMBOK Guideページ](https://www.pmi.org/standards/pmbok) で確認できます。

SG試験では、PMBOKの細かい版の違いよりも、次のような管理対象の切り分けを押さえることが大切です。

| 用語 | 判断基準 |
|---|---|
| [プロジェクト統合マネジメント](/sg/project-integration-management/) | 全体の整合性を調整する |
| [ステークホルダマネジメント](/sg/stakeholder-management/) | 関係者の期待・利害・影響を管理する |
| [スコープマネジメント](/sg/scope-management/) | 何をどこまで行うかを管理する |
| [タイムマネジメント](/sg/time-management/) | 作業順序・所要期間・日程を管理する |
| [コストマネジメント](/sg/cost-management/) | 費用・予算・実績を管理する |
| [リスクマネジメント](/sg/project-risk-management/) | まだ起きていない不確実な問題に備える |
| [品質マネジメント](/sg/quality-management/) | 成果物が求められる水準を満たすか管理する |
| [コミュニケーションマネジメント](/sg/communication-management/) | 誰に・何を・いつ・どう伝えるかを管理する |
| [調達マネジメント](/sg/procurement-management/) | 外部から必要なものを入手し、契約・納品を管理する |

この表で特に大切なのは、**用語名ではなく見るポイント**です。

問題文に「対象範囲」「追加機能」「どこまで行うか」とあれば、スコープマネジメントを考えます。

問題文に「作業順序」「所要日数」「納期」とあれば、タイムマネジメントを考えます。

問題文に「費用」「予算」「実績」とあれば、コストマネジメントを考えます。

## 作業分解とスケジュール管理

プロジェクトマネジメントでは、作業を洗い出し、順序を整理し、全体の所要期間を考えることも重要です。

SG試験では、WBS、アローダイアグラム、クリティカルパス、ダミー作業がセットで問われることがあります。

| 用語 | 判断基準 |
|---|---|
| [WBS](/sg/wbs/) | 作業を分解して管理しやすくする |
| [アローダイアグラム](/sg/arrow-diagram/) | 作業順序と所要日数を矢印で表す |
| [クリティカルパス](/sg/critical-path/) | 全体期間を決める最長経路 |
| [ダミー作業](/sg/dummy-activity/) | 依存関係だけを表す所要日数0の仮作業 |

WBSは、プロジェクトの作業を細かく分けるための考え方です。

アローダイアグラムは、作業の前後関係や所要日数を図で表すために使います。

クリティカルパスは、アローダイアグラムの中で、プロジェクト全体の期間を決める最長経路です。

ダミー作業は、実際の作業ではなく、依存関係を表すための仮の作業です。

SG試験では、次の3点を特に押さえておくと判断しやすくなります。

- 全作業を単純に足すのではなく、**経路ごとの合計日数**を見る
- クリティカルパスは、最短経路ではなく**最長経路**である
- ダミー作業は、**日数には足さないが、依存関係は読む**

## SG試験でのひっかけポイント

プロジェクトマネジメント分野では、似た用語の違いを問われることがあります。

選択肢を読むときは、次のように切り分けます。

| 迷いやすい組合せ | 切り分けのポイント |
|---|---|
| スコープ vs タイム | 範囲の話か、日程の話か |
| スコープ vs 統合 | 範囲そのものか、変更の全体影響か |
| タイム vs コスト | 期間の話か、費用の話か |
| リスク vs 課題 | 起きる前の備えか、起きた後の対応か |
| リスクマネジメント vs リスクアセスメント | 対応・見直しまで含むか、評価までか |
| ステークホルダ vs コミュニケーション | 関係者そのものか、情報共有の方法か |
| 調達 vs 委託先管理 | 外部から入手する話か、委託先を管理する話か |
| 品質 vs テスト | 品質全体の管理か、確認手段の1つか |
| アローダイアグラム vs クリティカルパス | 図全体か、図の中の最長経路か |
| ダミー作業 vs 実作業 | 依存関係だけか、実際に時間がかかる作業か |

SG試験では、「それっぽい説明」でも、対象が少しずれていることがあります。

たとえば、ステークホルダマネジメントは関係者の期待や影響を管理する話です。

一方、コミュニケーションマネジメントは、情報を誰に、いつ、どう伝えるかを管理する話です。

どちらも人に関係しますが、見るポイントが違います。

## まとめ（試験直前用）

- プロジェクトマネジメントは、目的に合った成果物を、限られた時間・費用・資源の中で完成させるための管理です。
- PMBOKは、プロジェクトマネジメントの知識や実務上の考え方を整理したガイドです。
- SG試験では、用語の丸暗記より、**何を管理する話か**を見分けることが大切です。
- WBSは作業分解、アローダイアグラムは作業順序、クリティカルパスは最長経路、ダミー作業は依存関係を表します。
- 選択肢では、範囲・日程・費用・品質・リスク・関係者・情報共有のどれを問われているかを確認します。
