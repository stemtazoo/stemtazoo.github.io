## まず結論

- このシリーズでは、暗号化・ハッシュ・MAC・電子署名・電子証明書を、**何を守る技術か**で整理します。
- SG試験では、**機密性**、**完全性**、**真正性**、**否認防止**のどれを実現したい場面かを判断できれば、選択肢を切りやすくなります。
- 丸暗記ではなく、**元に戻す必要があるか、鍵を使うか、公開鍵基盤が関係するか、失効確認が必要か**で切り分けます。

## 全体像

暗号・証明書・改ざん検知の用語は、すべて「安全な通信」や「データの正しさ」に関係します。  
ただし、試験で問われる役割は同じではありません。

- **暗号化**は、第三者に内容を読まれないようにする技術です。
- **ハッシュ**は、データが変わっていないかを確認するための要約値です。
- **MAC**は、共通鍵を使って改ざん検知と送信者確認を行います。
- **電子署名**は、秘密鍵と公開鍵を使って改ざん検知・本人確認・否認防止を行います。
- **電子証明書**は、公開鍵が本当にその相手のものかを認証局が証明します。
- **CRL / OCSP**は、発行済み証明書が今も有効かを確認する仕組みです。

SG試験では、細かい暗号アルゴリズムの計算よりも、**その技術が何を確認しているのか**が問われやすいです。  
「秘密にする」「変わっていないことを確認する」「相手や公開鍵の正当性を確認する」という順で整理すると、似た用語を見分けやすくなります。

## SG試験でのひっかけポイント

| 迷いやすい組合せ | 切り分けのポイント |
|---|---|
| 共通鍵暗号 vs 公開鍵暗号 | 同じ鍵を使うなら共通鍵、鍵ペアを使うなら公開鍵。速度と鍵配送の違いで判断する。 |
| 暗号化 vs ハッシュ | 暗号化は復号して元に戻す。ハッシュは元に戻さず、同一性や改ざん有無を見る。 |
| ハッシュ vs MAC | ハッシュだけでは送信者確認が弱い。MACは共通鍵を使うため、鍵を持つ相手からのものかを確認できる。 |
| MAC vs 電子署名 | MACは共有鍵なので第三者への証明が弱い。電子署名は公開鍵で検証でき、否認防止に強い。 |
| 電子署名 vs 電子証明書 | 電子署名はデータに対する処理。電子証明書は公開鍵の持ち主を証明するもの。 |
| 電子証明書 vs CRL / OCSP | 証明書は公開鍵の正当性を示す。CRL / OCSPはその証明書が失効していないかを確認する。 |
| CRL vs OCSP | CRLは失効リストを参照する。OCSPはオンラインで個別に有効性を問い合わせる。 |
| SSL/TLS vs 電子署名 | SSL/TLSは通信路を守る仕組み。電子署名は文書やデータの改ざん検知・本人証明に使う。 |

## まとめ（試験直前用）

- 機密性なら暗号化、完全性ならハッシュ・MAC・電子署名を疑う。
- ハッシュは元に戻さない。暗号化は復号して元に戻す。
- MACは共通鍵、電子署名は秘密鍵・公開鍵。否認防止が出たら電子署名を考える。
- 電子証明書は公開鍵の身元証明、CRL / OCSPは証明書の失効確認。
- 問題文では「復号」「共有鍵」「公開鍵」「失効」「否認防止」の語を拾って判断する。
