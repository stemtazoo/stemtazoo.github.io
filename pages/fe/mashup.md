## まず結論

マッシュアップとは、**公開されている複数のWebサービスやAPIを組み合わせて、新しいサービスを提供する考え方**です。

基本情報技術者試験では、次のように判断すると選択肢を切りやすいです。

```text
公開されている複数のサービスを利用する
↓
組み合わせて新しいサービスを作る
→ マッシュアップ
```

試験で **「公開されている複数のサービスを利用して、新たなサービスを提供する」** と説明されたら、マッシュアップを選びます。

## 直感的な説明

マッシュアップは、料理でいうと **既にある食材やソースを組み合わせて、新しい料理を作る** イメージです。

ITでは、外部のWebサービスやAPIを使って、新しい機能を作ります。

例えば、次のような組み合わせです。

```text
地図サービス
+
店舗情報サービス
+
口コミサービス
↓
周辺のお店を地図上に表示するサービス
```

このように、全部を一から作るのではなく、既存の公開サービスを組み合わせて別の価値を作るのがマッシュアップです。

## 定義・仕組み

マッシュアップは、複数のWebサービスやAPIを組み合わせて、新しいアプリケーションやサービスを作る開発手法です。

FE試験では、細かい実装よりも、次の対応を押さえると十分です。

| 用語 | 意味 |
|---|---|
| Webサービス | Web経由で利用できるサービス |
| Web API | 他のシステムから機能やデータを利用するための窓口 |
| マッシュアップ | 複数の公開サービスやAPIを組み合わせる考え方 |
| 新サービス | 組み合わせによって提供される新しい価値 |

たとえば、天気情報API、地図API、交通情報APIを組み合わせると、移動計画を支援するサービスを作れます。

```text
天気情報
+
地図情報
+
交通情報
↓
外出に便利な案内サービス
```

このテーマは、基本情報技術者試験の「ストラテジ系」や「システム活用」「Webサービス」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

## Web APIとの関係

マッシュアップでは、Web APIがよく使われます。

ただし、マッシュアップとWeb APIは同じ意味ではありません。

| 用語 | 位置づけ | ざっくり |
|---|---|---|
| Web API | 機能やデータを利用するための窓口 | 部品を呼び出す入口 |
| マッシュアップ | 複数のサービスを組み合わせる開発の考え方 | 組み合わせて新サービスを作る |
| JSON | APIでやり取りされるデータ形式の一つ | データの形 |

例えば、地図APIと店舗情報APIを使ってサービスを作る場合、Web APIはそれぞれの機能を使うための入口です。

マッシュアップは、それらを組み合わせて新しいサービスとして提供する考え方です。

```text
Web API
→ 使うための窓口

マッシュアップ
→ 複数の窓口を組み合わせて新サービスを作る
```

## 科目Aでどう出る？

科目Aでは、マッシュアップはリバースエンジニアリング、コンポーネント指向、オブジェクト指向などと並べて出題されやすいです。

判断するときは、**公開サービスを組み合わせているか** を見ます。

| 問題文の表現 | 選びたい用語 |
|---|---|
| 公開されている複数のサービスを利用して新サービスを提供する | マッシュアップ |
| 既存プログラムから仕様を導き出す | リバースエンジニアリング |
| 既存プログラムを部品化して組み合わせる | コンポーネント指向 |
| クラスライブラリを利用して開発する | オブジェクト指向・ライブラリ利用 |

似た選択肢は、次のように切ります。

| 選択肢の方向性 | 判断 |
|---|---|
| 既存のプログラムから仕様を導き出す | リバースエンジニアリング |
| 既存のプログラムを部品化して組み合わせる | コンポーネント指向 |
| クラスライブラリを利用して新規プログラムを開発する | ライブラリ利用・オブジェクト指向の文脈 |
| 公開されている複数のサービスを利用して新たなサービスを提供する | マッシュアップ |

決め手は、**公開されている複数のサービス** です。

```text
公開サービス + 組み合わせ + 新サービス
→ マッシュアップ
```

## 科目Bでどう使う？

科目Bでマッシュアップそのものを直接問う可能性は高くありません。

ただし、Webサービス連携やAPI利用の文章を読むときに、マッシュアップの考え方は役立ちます。

例えば、次のような事例です。

```text
外部の地図サービスを使う
外部の決済サービスを使う
自社の予約情報と組み合わせる
```

この場合、複数のサービスを連携させて、新しい利用体験を作っています。

科目Bや長文問題では、次のように整理すると読みやすいです。

```text
外部サービスを使っているか
↓
複数のサービスを組み合わせているか
↓
新しいサービスとして提供しているか
```

また、外部APIを利用する場合は、認証、可用性、利用制限、仕様変更などにも注意が必要です。

## よくある誤解・混同

マッシュアップで混同しやすいのは、リバースエンジニアリング、コンポーネント指向、Web APIです。

| 混同 | 正しい切り分け |
|---|---|
| マッシュアップは仕様を解析すること | 仕様を導き出すのはリバースエンジニアリング |
| マッシュアップはプログラム部品を組み合わせるだけ | 公開WebサービスやAPIを組み合わせる文脈で出やすい |
| マッシュアップとWeb APIは同じ | Web APIは窓口。マッシュアップは複数サービスを組み合わせる考え方 |
| マッシュアップはクラスライブラリ利用のこと | クラスライブラリ利用はプログラム開発の部品利用の文脈 |

特に、次の切り分けは重要です。

```text
既存プログラムから仕様を調べる
→ リバースエンジニアリング

プログラム部品を組み合わせる
→ コンポーネント指向

公開Webサービスを組み合わせる
→ マッシュアップ
```

## さらに注意したいひっかけ

マッシュアップでは、次のひっかけにも注意します。

| ひっかけ表現 | なぜ注意する？ |
|---|---|
| 既存プログラムから仕様を導く | リバースエンジニアリングの説明 |
| 既存プログラムを部品化する | コンポーネント指向の説明 |
| クラスライブラリを使う | ライブラリ利用・オブジェクト指向の説明 |
| 公開サービスを1つだけ単独で使う | マッシュアップでは複数サービスの組合せが中心 |
| サービスの仕様を逆解析する | マッシュアップではなく解析・調査の話 |

マッシュアップは、**外部の公開サービスを組み合わせて新しい価値を作る** ところがポイントです。

「組み合わせる」という言葉だけで判断せず、**何を組み合わせているか** を見ます。

## まとめ（試験直前用）

- マッシュアップは、公開されている複数のWebサービスやAPIを組み合わせて新サービスを提供する考え方
- 判断の合図は「公開サービス」「複数のサービス」「新たなサービス」
- 既存プログラムから仕様を導くのはリバースエンジニアリング
- 既存プログラムを部品化して組み合わせるのはコンポーネント指向
- クラスライブラリ利用は、マッシュアップとは別の開発文脈
- Web APIは利用する窓口、マッシュアップは組み合わせて新サービスを作る考え方
- 試験では「公開サービスを組み合わせるか」で切る
