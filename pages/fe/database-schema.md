## まず結論

スキーマとは、**データベースの構造や見方、格納方法を記述した定義**です。

基本情報技術者試験では、特に ANSI/SPARC の **3層スキーマ** が出題されやすいです。

| スキーマ | 何を表す？ | 判断の合図 |
|---|---|---|
| 外部スキーマ | 利用者から見たデータの見方 | 利用者ごとの見方、ビュー |
| 概念スキーマ | データベース全体の論理構造 | 業務データ、表、E-R図、正規化 |
| 内部スキーマ | 記録媒体上での格納方法 | ファイル編成、インデックス、物理格納 |
| サブスキーマ | 外部スキーマの別名として出ることがある | 外部スキーマの別称 |

今回の例題のように、**データベースを記録媒体にどのように格納するか** を記述するのは、**内部スキーマ** です。

## 直感的な説明

スキーマは、**データベースの設計図** のようなものです。

ただし、設計図にはいくつかの見方があります。

例えば、家で考えると分かりやすいです。

- 住む人から見た間取り
- 建物全体の設計図
- 配管や配線、基礎などの内部構造

データベースでも同じです。

```text
利用者からの見方      → 外部スキーマ
データベース全体構造  → 概念スキーマ
記録媒体への格納方法  → 内部スキーマ
```

この3つを分けることで、利用者の見え方を変えても、物理的な格納方法への影響を小さくできます。

## 定義・仕組み

ANSI/SPARC の3層スキーマは、データベースの定義を3つの層に分けて整理する考え方です。

FE試験では、細かい歴史よりも、次の対応を押さえることが大切です。

| 層 | 内容 | 例 |
|---|---|---|
| 外部スキーマ | 利用者やアプリケーションごとの見方 | ビュー、利用者別の表示項目 |
| 概念スキーマ | データベース全体の論理的な構造 | 表定義、E-R図、正規化、関係モデル |
| 内部スキーマ | データを記録媒体に格納する方法 | ファイル編成、インデックス、格納形式 |
| サブスキーマ | 外部スキーマの別称として出ることがある | 利用者側の部分的な見方 |

概念スキーマは、データベース化する業務やデータの内容を、論理的な構造として表します。

外部スキーマは、利用者やアプリケーションごとに必要なデータの見方を表します。

内部スキーマは、データを記憶装置上にどのような形式で格納するかを表します。

このテーマは、基本情報技術者試験の「データベース」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

## 科目Aでどう出る？

科目Aでは、スキーマは、説明文と用語の対応を選ぶ問題として出題されやすいです。

判断するときは、**利用者から見た話か、論理構造の話か、物理格納の話か** を見ます。

| 問題文の表現 | 選ぶ用語 |
|---|---|
| 利用者からの見方を記述する | 外部スキーマ |
| 利用者ごとのビューを記述する | 外部スキーマ |
| 業務データ全体を論理的に記述する | 概念スキーマ |
| E-R図、表定義、正規化に関係する | 概念スキーマ |
| 記録媒体にどのように格納するかを記述する | 内部スキーマ |
| ファイル編成やインデックスに関係する | 内部スキーマ |
| 外部スキーマの別称 | サブスキーマ |

例えば、次のような説明は内部スキーマです。

```text
データベースを記録媒体にどのように格納するかを記述する。
```

この説明では、**記録媒体にどのように格納するか** が決め手です。

一方で、次のような説明は概念スキーマです。

```text
データベース化対象の業務とデータの内容を，
論理的な構造として記述する。
```

こちらは、業務データ全体を論理的に表すため、概念スキーマです。

科目Aでは、**利用者なら外部、論理構造なら概念、記録媒体なら内部** と考えると選択肢を切りやすいです。

## 科目Bでどう使う？

スキーマは、科目Bの擬似言語トレース問題で直接計算するテーマではありません。

ただし、データベース設計やシステム設計の説明を読むときに、スキーマの考え方は役立ちます。

例えば、次のような場面を考えます。

```text
営業担当者には顧客名と連絡先だけを見せる。
経理担当者には請求先と支払状況を見せる。
```

このような利用者ごとの見え方は、外部スキーマに近い考え方です。

一方で、次のような場面は概念スキーマに近いです。

```text
顧客、商品、注文、請求の関係を整理し，
テーブル構造として設計する。
```

さらに、次のような場面は内部スキーマに近いです。

```text
検索を速くするためにインデックスを設定する。
データをどのファイル形式で格納するかを決める。
```

科目Bや長文問題で読むときは、次の順で整理すると分かりやすいです。

1. 説明が利用者からの見方かを見る
2. データベース全体の論理構造かを見る
3. 記録媒体への格納方法かを見る
4. どの層のスキーマの話かを判断する

スキーマは、単なる「表」ではありません。  
データをどう見せるか、どう整理するか、どう保存するかを分けて考えるための定義です。

## SG試験との違い

スキーマは、情報セキュリティマネジメント試験では、FEほど主役になりにくい用語です。

ただし、データベースのアクセス制御やデータ管理を考えるときに、外部スキーマの考え方は関係します。

| 試験 | 見られやすいポイント |
|---|---|
| FE | 3層スキーマの名称と役割の切り分け |
| SG | 利用者に必要なデータだけを見せる、アクセス権限を管理する |
| 共通 | データの見方や構造を分けて考える |

FEでは、**外部・概念・内部** の対応を問われやすいです。

SGでは、用語そのものよりも、利用者ごとに見せる情報を制限する、不要なデータを見せない、という運用面の考え方につながります。

## よくある誤解・混同

スキーマで混同しやすいのは、概念スキーマ、外部スキーマ、内部スキーマ、サブスキーマです。

| 用語 | 意味 | 判断の合図 |
|---|---|---|
| 外部スキーマ | 利用者から見たデータの見方 | 利用者、ビュー、表示項目 |
| 概念スキーマ | データベース全体の論理構造 | 業務データ、E-R図、表定義、正規化 |
| 内部スキーマ | 記録媒体上での物理的な格納方法 | 記録媒体、ファイル編成、インデックス |
| サブスキーマ | 外部スキーマの別名として出ることがある | 外部スキーマの別称 |

特に、概念スキーマと内部スキーマは混同しやすいです。

- 概念スキーマ：データの意味や関係をどう整理するか
- 内部スキーマ：データを記録媒体にどう格納するか

また、外部スキーマと概念スキーマも混同しやすいです。

- 外部スキーマ：利用者ごとの見え方
- 概念スキーマ：データベース全体の論理構造

試験では、**誰から見た話か** と **どのレベルの話か** を見ると判断しやすくなります。

## さらに注意したいひっかけ

スキーマでは、次のようなひっかけにも注意します。

| ひっかけ表現 | なぜ注意する？ |
|---|---|
| 記録媒体に格納する方法を概念スキーマとする | 物理格納は内部スキーマ |
| 利用者ごとの見方を概念スキーマとする | 利用者側の見方は外部スキーマ |
| E-R図や正規化を内部スキーマとする | 論理構造は概念スキーマ |
| サブスキーマを独立した第4層と考える | サブスキーマは外部スキーマの別称として出やすい |
| インデックスを概念スキーマとする | インデックスは内部スキーマ側の話になりやすい |
| ビューを内部スキーマとする | ビューは外部スキーマ側の話になりやすい |

今回の例題では、**データベースを記録媒体にどのように格納するか** という表現が出ています。

この場合、記録媒体という言葉があるため、内部スキーマを選びます。

逆に、問題文に **利用者からの見方** があれば外部スキーマ、**業務データの論理構造** があれば概念スキーマを選びます。

## まとめ（試験直前用）

- スキーマは、データベースの構造や見方、格納方法を表す定義
- 利用者からの見方は外部スキーマ
- 業務データ全体の論理構造は概念スキーマ
- 記録媒体への格納方法は内部スキーマ
- サブスキーマは外部スキーマの別称として出やすい
- 試験では「利用者」「論理構造」「記録媒体」のどれかを先に見る
