## まず結論

データベース正規化とは、**データの重複や不整合を減らすために、表を適切に分ける考え方**です。

基本情報技術者試験では、まず第3正規形までを判断できれば十分です。

```text
第1正規形
→ 繰り返し項目をなくす

第2正規形
→ 主キーの一部だけで決まる項目を分ける

第3正規形
→ 主キー以外で決まる項目を分ける
```

試験では、難しい理論よりも、**どの項目が何によって決まるか** を見るのが大切です。

## 直感的な説明

正規化は、表の中に混ざっている情報を整理する作業です。

例えば、発注伝票の表があるとします。

```text
発注伝票(注文番号, 商品番号, 商品名, 注文数量)
```

このとき、主キーが `注文番号 + 商品番号` だとします。

注文数量は、どの注文で、どの商品を、いくつ注文したかで決まります。

```text
注文番号 + 商品番号 → 注文数量
```

一方で、商品名は商品番号だけで決まります。

```text
商品番号 → 商品名
```

つまり、商品名は発注伝票の中に置いておくより、商品表に分けた方が自然です。

```text
発注(注文番号, 商品番号, 注文数量)
商品(商品番号, 商品名)
```

このように、何によって決まる項目かを見ながら表を分けるのが正規化です。

## 定義・仕組み

正規化では、関数従属という考え方が出てきます。

関数従属とは、ある項目が決まると、別の項目も一意に決まる関係です。

```text
商品番号 → 商品名
```

これは、商品番号が決まれば商品名も決まる、という意味です。

FE試験では、関数従属を厳密に証明するより、表の分け方を判断するために使います。

| 正規形 | 見るポイント | 例 |
|---|---|---|
| 第1正規形 | 1つの欄に複数値が入っていないか | 電話番号1、電話番号2のような繰り返し |
| 第2正規形 | 複合主キーの一部だけで決まる項目がないか | 商品番号 → 商品名 |
| 第3正規形 | 主キー以外で決まる項目がないか | 部署番号 → 部署名 |

第2正規形では、複合主キーの一部だけで決まる項目を別表に分けます。

```text
注文番号 + 商品番号 が主キー

商品番号だけで商品名が決まる
↓
商品名は商品表へ分ける
```

第3正規形では、主キー以外の項目によって決まる項目を別表に分けます。

```text
社員番号 → 部署番号 → 部署名

部署名は社員番号から直接決まるのではなく、
部署番号によって決まる
↓
部署表へ分ける
```

このテーマは、基本情報技術者試験の「データベース」や「データベース設計」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

## 科目Aでどう出る？

科目Aでは、表を第3正規形に分解した結果を選ぶ問題として出題されやすいです。

今回のような表を考えます。

```text
発注伝票(注文番号, 商品番号, 商品名, 注文数量)
```

主キーが、

```text
注文番号 + 商品番号
```

である場合、まず各項目が何で決まるかを見ます。

```text
注文番号 + 商品番号 → 注文数量
商品番号 → 商品名
```

商品名は、複合主キー全体ではなく、主キーの一部である商品番号だけで決まります。

これは第2正規形の条件に引っかかります。

そのため、次のように分けます。

```text
発注(注文番号, 商品番号, 注文数量)
商品(商品番号, 商品名)
```

発注表には、商品表を参照するために `商品番号` を残します。

ここで `商品番号` を発注表から消してしまうと、どの商品を注文したのか分からなくなります。

## 科目Bでどう使う？

科目Bでは、正規化そのものの名前を問うより、データの重複や更新ミスを避ける設計として出てくることがあります。

例えば、商品名を発注表に毎回入れていると、同じ商品名が何度も出てきます。

```text
注文001, 商品A, ねじ, 10
注文002, 商品A, ねじ, 5
注文003, 商品A, ネジ, 7
```

このように、表記ゆれや更新漏れが起きやすくなります。

商品表に分けておけば、商品名を管理する場所は1か所になります。

```text
商品(商品番号, 商品名)
```

発注表は商品番号だけを持てばよくなります。

```text
発注(注文番号, 商品番号, 注文数量)
```

科目Bでは、次のように読むと判断しやすいです。

```text
同じ情報が何度も出ている
↓
更新ミスが起きそう
↓
何で決まる項目かを見る
↓
別表に分ける
```

## よくある誤解・混同

正規化でよくある誤解は、**主キーに含まれる項目を、分解後の表から消してしまうこと** です。

| 誤解 | 正しい理解 |
|---|---|
| 商品名を商品表に分けたら、発注表から商品番号も消す | 発注表には商品表を参照する商品番号を残す |
| 第2正規形は、主キー以外で決まる項目を分ける | それは第3正規形の考え方 |
| 第3正規形では、複合主キーの一部で決まる項目を見る | それは第2正規形の考え方 |
| 正規化は表を細かくするほどよい | 必要以上に分けると扱いにくくなることもある |
| FEでは第4正規形以降まで深く必要 | 基本は第3正規形までを判断できればよい |

試験では、次の切り分けを残しておくと強いです。

```text
繰り返しをなくす
→ 第1正規形

主キーの一部で決まる項目を分ける
→ 第2正規形

主キー以外で決まる項目を分ける
→ 第3正規形
```

特に、複合主キーの場合は、主キーの一部だけで決まる項目がないかを確認します。

## まとめ（試験直前用）

- 正規化は、重複や不整合を減らすために表を分ける考え方
- FEでは、第1〜第3正規形までを判断できれば十分
- 第1正規形は、繰り返し項目をなくす
- 第2正規形は、主キーの一部だけで決まる項目を分ける
- 第3正規形は、主キー以外で決まる項目を分ける
- 複合主キーでは、部分的に決まる項目に注意する
- 分解後も、参照に必要なキーは元の表に残す
