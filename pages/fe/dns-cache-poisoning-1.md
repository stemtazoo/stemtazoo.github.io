## まず結論

DNSキャッシュポイズニングとは、**DNSサーバのキャッシュに偽の名前解決情報を記憶させ、利用者を偽サイトへ誘導する攻撃**です。

基本情報技術者試験では、DNSキャッシュポイズニングは **正しいドメイン名を入力しても、偽のIPアドレスへ誘導される攻撃** として判断します。

ポイントは、利用者がURLを間違えたのではなく、**DNSの名前解決結果が偽情報にすり替わっている** ことです。

## 直感的な説明

DNSキャッシュポイズニングは、**電話帳に偽の電話番号を書き込まれる** イメージです。

本来、DNSは次のように名前とIPアドレスを対応付けます。

```text
www.example.com
↓
正しいIPアドレス
```

ところが、DNSキャッシュポイズニングでは、DNSサーバのキャッシュに偽の情報を入れられます。

```text
www.example.com
↓
攻撃者が用意した偽サイトのIPアドレス
```

利用者は正しいドメイン名を入力しているつもりでも、DNSが偽のIPアドレスを返すと、偽サイトへ誘導されてしまいます。

つまり、DNSキャッシュポイズニングは、**名前解決の結果をだまして、利用者の行き先を変える攻撃** です。

## 定義・仕組み

DNSキャッシュポイズニングは、DNSサーバが一時的に保存している名前解決情報、つまりDNSキャッシュに、不正な対応情報を記憶させる攻撃です。

DNSサーバは、問い合わせのたびに毎回最初から調べると時間がかかるため、一度得た結果をしばらくキャッシュします。

攻撃者は、このキャッシュに偽の情報を入れようとします。

基本的な流れは次のとおりです。

1. 利用者やDNSサーバが、あるドメイン名のIPアドレスを問い合わせる
2. 攻撃者が偽の応答を送り込む
3. DNSサーバが偽の応答を正しい情報としてキャッシュする
4. その後の利用者は、偽のIPアドレスへ誘導される

FE試験では、細かい攻撃手順よりも、次の対応を押さえると十分です。

| 用語 | 意味 |
|---|---|
| DNS | ドメイン名とIPアドレスを対応付ける仕組み |
| 名前解決 | ドメイン名からIPアドレスを調べること |
| DNSキャッシュ | 過去の名前解決結果を一時的に保存したもの |
| ポイズニング | 偽情報を混入させること |
| DNSキャッシュポイズニング | 偽の名前解決情報をキャッシュさせる攻撃 |

このテーマは、基本情報技術者試験の「情報セキュリティ」や「ネットワーク」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

## 科目Aでどう出る？

科目Aでは、DNSキャッシュポイズニングは、DNS、ファーミング、フィッシング、DNSSECなどと並べて出題されやすいです。

判断するときは、**DNSキャッシュに偽情報を入れているか** を見ます。

| キーワード | 判断 |
|---|---|
| DNSキャッシュに偽情報を記憶させる | DNSキャッシュポイズニング |
| 正しいドメイン名でも偽サイトへ誘導される | DNSキャッシュポイズニング |
| 名前解決結果を改ざんする | DNSキャッシュポイズニング |
| メールや偽サイトで利用者をだます | フィッシング |
| DNSを悪用して偽サイトへ誘導する広い攻撃 | ファーミング |
| DNS応答の正当性を検証する | DNSSEC |

例えば、次のような説明はDNSキャッシュポイズニングです。

```text
DNSサーバのキャッシュに偽の名前解決情報を記憶させ，
正しいドメイン名を入力した利用者を偽サイトへ誘導する。
```

この説明では、**DNSサーバのキャッシュ** と **偽の名前解決情報** が決め手です。

一方で、次のような説明はDNSキャッシュポイズニングではありません。

```text
利用者に偽メールを送り，偽サイトへアクセスさせて，
IDやパスワードを入力させる。
```

これは、利用者をだますフィッシングの説明です。

科目Aでは、**DNSキャッシュを汚すならDNSキャッシュポイズニング、利用者をメールでだますならフィッシング** と考えると選択肢を切りやすいです。

## 科目Bでどう使う？

DNSキャッシュポイズニングは、科目Bの擬似言語トレース問題で中心になることは多くありません。

ただし、ネットワーク障害やセキュリティ事故の説明を読むときに、DNSキャッシュポイズニングの考え方は役立ちます。

例えば、次のような状況を考えます。

```text
利用者は正しいURLを入力している。
しかし、別の偽サイトに接続されてしまう。
複数の利用者で同じ現象が起きている。
```

この場合、利用者の入力ミスだけでは説明しにくいです。

DNSの名前解決結果が誤っている可能性があります。

科目Bや長文問題で読むときは、次の順で整理すると分かりやすいです。

1. 利用者が正しいドメイン名を使っているかを見る
2. 接続先のIPアドレスが想定と違うかを見る
3. DNSキャッシュやDNS設定が関係しているかを見る
4. 偽サイト誘導や認証情報の窃取につながるかを見る
5. DNSSECなどの対策が使われているかを見る

DNSキャッシュポイズニングは、通信内容を盗聴する攻撃そのものではありません。  
名前解決をだまして、利用者を攻撃者の用意した場所へ誘導する攻撃です。

## SG試験との違い

DNSキャッシュポイズニングは、情報セキュリティマネジメント試験でも出やすい用語です。

FEとSGでは、見られやすいポイントが少し違います。

| 試験 | 見られやすいポイント |
|---|---|
| FE | DNS、DNSキャッシュ、DNSSEC、フィッシングとの機能の切り分け |
| SG | 偽サイト誘導、認証情報の漏えい、DNS設定管理、利用者影響 |
| 共通 | DNSの名前解決結果をだまして偽サイトへ誘導する攻撃 |

FEでは、技術的に「どの仕組みを悪用しているか」を問われやすいです。

SGでは、攻撃によって利用者が偽サイトに誘導される、IDやパスワードを入力してしまう、業務システムへ接続できなくなる、といった運用面が重視されやすいです。

どちらの試験でも、まずは **DNSキャッシュポイズニング＝DNSキャッシュに偽情報を入れる攻撃** と押さえるのが基本です。

## よくある誤解・混同

DNSキャッシュポイズニングで混同しやすいのは、フィッシング、ファーミング、DNSSEC、DNSハイジャックです。

| 用語 | 主な意味 | 判断の合図 |
|---|---|---|
| DNSキャッシュポイズニング | DNSキャッシュに偽情報を記憶させる攻撃 | キャッシュ、偽の名前解決、偽IP |
| フィッシング | 偽メールや偽サイトで利用者をだます攻撃 | メール、偽サイト、ID・パスワード入力 |
| ファーミング | 利用者を偽サイトへ誘導する攻撃の広い考え方 | 正しいURLでも偽サイト、誘導 |
| DNSSEC | DNS応答の正当性を検証する仕組み | 署名、検証、DNS応答の真正性 |
| DNSハイジャック | DNS設定やドメイン管理を乗っ取る攻撃 | 設定変更、ドメイン管理、乗っ取り |

フィッシングは、利用者をだまして偽サイトへ誘導する攻撃です。

DNSキャッシュポイズニングは、利用者が正しいドメイン名を使っていても、DNSの応答が偽情報になっているため、偽サイトへ誘導される点が特徴です。

ファーミングは、偽サイトへ誘導する攻撃を広く指す言葉として出ることがあります。  
DNSキャッシュポイズニングは、その手段の一つとして整理すると分かりやすいです。

DNSSECは、DNS応答が正当なものかを検証するための仕組みです。  
攻撃ではなく、DNSキャッシュポイズニングなどへの対策として出やすいです。

## さらに注意したいひっかけ

DNSキャッシュポイズニングでは、次のようなひっかけにも注意します。

| ひっかけ表現 | なぜ注意する？ |
|---|---|
| 利用者がURLを打ち間違える | タイポスクワッティングなどの説明になりやすい |
| 偽メールで偽サイトへ誘導する | フィッシングの説明 |
| DNS応答の正当性を検証する | DNSSECの説明 |
| IPアドレスを自動で割り当てる | DHCPの説明 |
| プライベートIPをグローバルIPに変換する | NATの説明 |
| 通信内容を暗号化する | HTTPSやVPNの説明 |
| DNSキャッシュを削除するだけで完全対策 | 根本対策には設定・検証・運用管理も必要 |

特に、**DNSキャッシュポイズニングは利用者の入力ミスではない** という点は重要です。

利用者は正しいドメイン名を入力していても、名前解決結果が偽情報になっていれば、偽サイトへ誘導されます。

また、HTTPSで通信している場合でも、証明書警告を無視してしまうと危険です。  
DNSの問題だけでなく、証明書確認や利用者教育も関係します。

## まとめ（試験直前用）

- DNSキャッシュポイズニングは、DNSキャッシュに偽の名前解決情報を記憶させる攻撃
- 正しいドメイン名でも、偽のIPアドレスへ誘導されることがある
- DNSは名前解決、DNSキャッシュポイズニングはその結果をだます攻撃
- 偽メールでだますならフィッシング
- DNS応答を検証する対策はDNSSEC
- FEでは技術の切り分け、SGでは偽サイト誘導や運用リスクを意識する
