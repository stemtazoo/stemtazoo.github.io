## まず結論

リンカ（Linker）とは、**複数の目的モジュールやライブラリをつないで、実行できる形のロードモジュールを作るソフトウェア**です。

英語の `link` は「つなぐ」という意味です。  
そのため、Linker は **つなぐもの** と考えると覚えやすいです。

基本情報技術者試験では、次のように切り分けると選択肢を選びやすくなります。

```text
Compiler：ソースコードを目的モジュールに変換する
Linker  ：目的モジュールやライブラリをつなぐ
Loader  ：ロードモジュールを主記憶に読み込む
Debugger：実行を確認して不具合を調べる
```

「相互参照の解決」や「複数の目的モジュールからロードモジュールを生成」と出たら、**リンカ（Linker）** を疑います。

## 直感的な説明

プログラムは、最初から1つの完成品として作られるとは限りません。

実際には、いくつかの部品に分けて作られることがあります。

```text
部品A
部品B
ライブラリ
```

リンカ（Linker）は、これらの部品をつないで、実行できる形にまとめる役割です。

たとえるなら、コンパイラ（Compiler）が部品を作り、リンカ（Linker）が部品を組み合わせて完成品にするイメージです。

```text
ソースコード
→ Compiler
→ 目的モジュール
→ Linker
→ ロードモジュール
→ Loader
→ 主記憶に読み込んで実行
```

この流れを押さえると、リンカとローダを混同しにくくなります。

## 定義・仕組み

プログラムが実行されるまでの流れを、ざっくり整理すると次のようになります。

| 段階 | 担当するもの | 役割 |
|---|---|---|
| 1 | コンパイラ（Compiler） | ソースコードを目的モジュールに変換する |
| 2 | リンカ（Linker） | 目的モジュールやライブラリを結合する |
| 3 | ローダ（Loader） | ロードモジュールを主記憶に読み込む |
| 4 | デバッガ（Debugger） | 実行を確認し、不具合を調べる |

リンカ（Linker）が行う重要な処理の一つが、**相互参照の解決** です。

例えば、ある目的モジュールが、別の目的モジュールにある関数やデータを使っている場合があります。

```text
目的モジュールA → 目的モジュールBの関数を参照
```

このような参照関係を解決し、複数の目的モジュールやライブラリをまとめて、実行できる形にします。

この実行できる形のプログラムを、ロードモジュールと呼ぶことがあります。

```text
目的モジュール + ライブラリ
→ Linker
→ ロードモジュール
```

公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。リンカ（Linker）は、プログラムの作成から実行までの流れを理解するうえで重要な用語です。

## 科目Aでどう出る？

科目Aでは、説明文から該当するソフトウェアを選ぶ形で出題されやすいです。

選択肢を切るときは、次のキーワードを見ます。

| 問題文のキーワード | 選ぶ用語 |
|---|---|
| ソースコードを変換する | コンパイラ（Compiler） |
| 目的モジュールを結合する | リンカ（Linker） |
| 相互参照を解決する | リンカ（Linker） |
| ロードモジュールを主記憶に読み込む | ローダ（Loader） |
| ステップ実行、実行結果を確認する | デバッガ（Debugger） |

例えば、

```text
相互参照の解決などを行い、
複数の目的モジュールなどから一つのロードモジュールを生成する
```

という説明なら、ポイントは **相互参照の解決** と **ロードモジュールを生成** です。

これはリンカ（Linker）の機能です。

一方、

```text
実行に先立ってロードモジュールを主記憶にロードする
```

なら、ローダ（Loader）です。

## 科目Bでどう使う？

科目Bでリンカ（Linker）そのものを細かく操作する問題は多くありません。

ただし、プログラムの作成から実行までの流れを読む場面では役立ちます。

例えば、エラーや処理の説明で、

- ソースコード
- 目的モジュール
- ライブラリ
- ロードモジュール
- 主記憶

のような言葉が出たとき、どの段階の話かを確認します。

```text
変換の話 → Compiler
結合の話 → Linker
読み込みの話 → Loader
実行確認の話 → Debugger
```

この流れが分かると、選択肢の説明がどの用語に対応しているかを読み取りやすくなります。

## よくある誤解・混同

一番混同しやすいのは、リンカ（Linker）とローダ（Loader）です。

| 混同しやすい組み合わせ | 違い |
|---|---|
| リンカ（Linker）とローダ（Loader） | Linkerは結合する、Loaderは主記憶に読み込む |
| コンパイラ（Compiler）とリンカ（Linker） | Compilerは変換する、Linkerはつなぐ |
| リンカ（Linker）とデバッガ（Debugger） | Linkerは実行前に結合する、Debuggerは実行を確認する |

覚え方としては、英語を見ると分かりやすいです。

```text
compile：変換してまとめる
link   ：つなぐ
load   ：読み込む
debug  ：バグを取り除く・調べる
```

特に、Linker は `link` から考えると、**部品をつなぐもの** とイメージできます。

また、ローダ（Loader）は、ロードモジュールを作るのではなく、主記憶に読み込みます。

```text
ロードモジュールを作る → Linker
ロードモジュールを読み込む → Loader
```

この違いを押さえると、選択肢を切りやすくなります。

## まとめ（試験直前用）

- リンカ（Linker）は、目的モジュールやライブラリを **つなぐ** ソフトウェア
- Compilerは、ソースコードを目的モジュールに変換する
- Linkerは、相互参照を解決し、ロードモジュールを生成する
- Loaderは、ロードモジュールを主記憶に読み込む
- Debuggerは、プログラムの実行を確認して不具合を調べる
