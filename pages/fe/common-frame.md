## まず結論

共通フレームとは、**システムやソフトウェアに関わる作業を、共通のプロセスとして整理するための枠組み**です。

基本情報技術者試験では、共通フレームを **企画、要件定義、開発、運用、保守、支援などの作業を共通の言葉で整理したもの** と考えると分かりやすいです。

古い過去問では「共通フレーム2007」と書かれていることがあります。学習記事では、現在の学習向けに **共通フレーム2013の考え方にも合うように**、細かい版の暗記よりも、プロセスの目的で切り分けます。

特に、企画プロセスは次のように押さえます。

```text
経営事業の目的・目標を達成するために、
必要なシステム化の方針と実施計画を得る
→ 企画プロセス
```

試験では、**「経営目的・目標」＋「システム化方針」＋「実施計画」** が出たら、企画プロセスを疑います。

## 直感的な説明

共通フレームは、システム開発のための **共通の地図** のようなものです。

システム開発には、利用者、発注者、開発者、運用担当者、保守担当者など、いろいろな立場の人が関わります。

それぞれが別々の言葉で作業を説明すると、認識がずれやすくなります。

```text
経営層：事業目標を達成したい
業務部門：業務をこう変えたい
情報システム部門：システムとして実現したい
開発者：仕様に沿って作りたい
運用担当：安定して使えるようにしたい
```

共通フレームでは、こうした作業を共通のプロセスとして整理します。

つまり、共通フレームは **関係者が同じ地図を見ながら、どの作業の話をしているかを合わせるための枠組み** です。

## 定義・仕組み

共通フレームでは、システムやソフトウェアのライフサイクルに関わる作業を、プロセスとして整理します。

FE試験では、全体像を次のように押さえると十分です。

| プロセス群・視点 | 役割のイメージ |
|---|---|
| 合意プロセス | 取得や供給など、契約・合意に関わる |
| 企画プロセス | 経営目的を達成するためのシステム化方針と実施計画を得る |
| 要件定義プロセス | 利用者や業務から見た要求を明確にする |
| 開発プロセス | システムやソフトウェアを設計・実装・テストする |
| 運用プロセス | 稼働したシステムを使い続けられるようにする |
| 保守プロセス | 稼働後の変更や修正に対応する |
| マネジメントプロセス | プロジェクト、品質、リスクなどを管理する |
| 支援プロセス | 品質保証、検証、妥当性確認、文書化、問題解決などで活動を支える |

共通フレームのポイントは、具体的な開発手法そのものではなく、**作業をどのようなプロセスとして整理するか** にあります。

このテーマは、基本情報技術者試験の「ソフトウェア開発」や「システム開発技術」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

共通フレームは、国際規格のシステムライフサイクルプロセスやソフトウェアライフサイクルプロセスの考え方と関係します。一次情報としては、[ISO：ISO/IEC/IEEE 15288:2023](https://www.iso.org/standard/81702.html) や [ISO：ISO/IEC/IEEE 12207:2017](https://www.iso.org/standard/63712.html) も確認先になります。

## 科目Aでどう出る？

科目Aでは、共通フレームそのものの意味や、プロセス名と説明の対応を問う形で出題されやすいです。

判断するときは、プロセス名の細部よりも、まず **何をしている活動か** を見ます。

| 問題文・選択肢の表現 | 見る方向 |
|---|---|
| 経営事業の目的・目標、システム化の方針、実施計画 | 企画プロセス |
| 利用者の要求を明確にする、業務要件を整理する | 要件定義プロセス |
| 設計、実装、テストを行う | 開発プロセス |
| 稼働後にシステムを使い続ける | 運用プロセス |
| 稼働後の変更・修正に対応する | 保守プロセス |
| 作業成果物やプロセスが条件・計画に従っていることを保証する | 品質保証プロセス |
| 発見された問題を識別・分析・管理・制御し、解決する | 問題解決プロセス |
| 生成・記録された情報を文書化し、保守する | 文書化プロセス |
| 仕様どおりか確認する | 検証プロセス |
| 利用者の目的に合っているか確認する | 妥当性確認プロセス |

今回のような選択肢では、次のように切ります。

| 選択肢の内容 | 対応するプロセス |
|---|---|
| 経営事業の目的・目標を達成するため、システム化方針と実施計画を得る | 企画プロセス |
| 成果物やプロセスが条件・計画に従っていることを保証する | 品質保証プロセス |
| 問題を識別、分析、管理、制御し、解決する | 問題解決プロセス |
| システムやソフトウェア情報を文書化し、保守する | 文書化プロセス |

共通フレーム全体の用語を暗記するより、このように **目的語で切る** 方が実戦的です。

```text
方針・計画を得る → 企画
条件・計画どおりを保証 → 品質保証
問題を解決 → 問題解決
情報を文書化 → 文書化
```

## 科目Bでどう使う？

科目Bで共通フレームのプロセス名だけを直接問う場面は多くありません。

ただし、システム開発の長い説明文を読むときに、**今どの段階の話をしているか** を整理する力は役立ちます。

例えば、次のように読み分けます。

```text
経営目標からシステム化方針を作っている
→ 企画寄り

業務部門の要求を整理している
→ 要件定義寄り

設計・実装・テストしている
→ 開発寄り

稼働後に利用・監視している
→ 運用寄り

稼働後に修正・変更している
→ 保守寄り

正しさを確認している
→ 支援プロセス寄り
```

科目Bでは、問題文の登場人物や工程が多くなることがあります。

そのときに、共通フレームの考え方を使って、**これは企画の話か、要件定義の話か、開発の話か、運用・保守の話か、確認・管理の話か** を分けると読みやすくなります。

## よくある誤解・混同

共通フレームでよくある誤解は、**開発手法の一つだと思ってしまうこと** です。

| 誤解 | 正しい理解 |
|---|---|
| 共通フレームはウォーターフォールやアジャイルの一種 | 開発手法ではなく、作業を共通のプロセスとして整理する枠組み |
| 共通フレームは開発者だけが使う | 発注者、利用者、開発者、運用担当など関係者全体に関わる |
| 企画プロセスは要件定義と同じ | 企画は経営目的からシステム化方針・実施計画を得る。要件定義は要求を明確にする |
| 品質保証はテストそのもの | 成果物やプロセスが計画・基準に従っていることを保証する活動 |
| 問題解決は開発そのもの | 発見された問題を識別・分析・管理・制御し、解決する活動 |
| 文書化は設計だけの作業 | プロセスで生成・記録された情報を文書化し、保守する活動 |
| 検証と妥当性確認は同じ | 検証は仕様どおり、妥当性確認は利用者の目的どおり |

特に、企画プロセスと要件定義プロセスは近く見えます。

- 企画プロセス：**経営目的から、システム化方針と実施計画を得る**
- 要件定義プロセス：**利用者や業務から見た要求を明確にする**

試験では、**経営目的・目標** が出てきたら、企画プロセス側に寄せて考えると判断しやすいです。

## まとめ（試験直前用）

- 共通フレームは、システム開発や運用の作業を共通のプロセスで整理する枠組み
- 古い過去問で「共通フレーム2007」と出ても、現在は共通フレーム2013の考え方にも合うように、プロセスの目的で切る
- 企画プロセスは、経営目的・目標を達成するためのシステム化方針と実施計画を得る
- 品質保証プロセスは、成果物やプロセスが条件・計画に従っていることを保証する
- 問題解決プロセスは、発見された問題を識別・分析・管理・制御し、解決する
- 文書化プロセスは、生成・記録された情報を文書化し、保守する
- 「仕様どおりか」は検証、「利用者の目的どおりか」は妥当性確認
