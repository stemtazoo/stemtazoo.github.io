## まず結論

共通フレームとは、**システムやソフトウェアに関わる作業を、共通のプロセスとして整理するための枠組み**です。

基本情報技術者試験では、共通フレームを **企画、開発、運用、保守、支援などの作業を共通の言葉で整理したもの** と考えると分かりやすいです。

細かいプロセス名をすべて暗記するよりも、まずは **どのプロセス群の話か** を切り分けることが大切です。

## 直感的な説明

共通フレームは、システム開発のための **共通の地図** のようなものです。

システム開発には、利用者、発注者、開発者、運用担当者、保守担当者など、いろいろな立場の人が関わります。

それぞれが別々の言葉で作業を説明すると、認識がずれやすくなります。

```text
発注者：この作業は依頼の範囲です
開発者：これは開発の範囲です
運用担当：これは運用後の対応です
```

共通フレームでは、こうした作業を共通のプロセスとして整理します。

つまり、共通フレームは **関係者が同じ地図を見ながら、どの作業の話をしているかを合わせるための枠組み** です。

## 定義・仕組み

共通フレームでは、システムやソフトウェアのライフサイクルに関わる作業を、プロセスとして整理します。

FE試験では、全体像を次のように押さえると十分です。

| プロセス群 | 役割のイメージ |
|---|---|
| 合意プロセス | 取得や供給など、契約・合意に関わる |
| 企画・要件定義の視点 | 何を作るか、何が必要かを整理する |
| 開発・保守の視点 | システムやソフトウェアを作る、直す |
| 運用・サービスの視点 | 作ったシステムを運用し、サービスとして提供する |
| マネジメントプロセス | プロジェクトや品質、リスクなどを管理する |
| 支援プロセス | 検証、妥当性確認、監査などで活動を支える |

共通フレームのポイントは、具体的な開発手法そのものではなく、**作業をどのようなプロセスとして整理するか** にあります。

例えば、検証プロセスや妥当性確認プロセスは、開発作業そのものというより、成果物やサービスが正しいかを確認する支援的な活動です。

このテーマは、基本情報技術者試験の「ソフトウェア開発」や「システム開発技術」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

## 科目Aでどう出る？

科目Aでは、共通フレームそのものの意味や、プロセス名と説明の対応を問う形で出題されやすいです。

判断するときは、プロセス名の細部よりも、まず **何をしている活動か** を見ます。

| 問題文・選択肢の表現 | 見る方向 |
|---|---|
| 契約、取得、供給 | 合意プロセス |
| 何を作るか、要求を整理する | 企画・要件定義の視点 |
| システムを作る、ソフトウェアを実装する | 開発の視点 |
| 運用する、サービスを提供する | 運用・サービスの視点 |
| プロジェクトを管理する | マネジメントプロセス |
| 検証、妥当性確認、監査を行う | 支援プロセス |

特に、共通フレームの問題では、**支援プロセス** が選択肢に出ることがあります。

その場合は、次の切り分けが重要です。

```text
仕様どおりか → 検証
利用者の目的どおりか → 妥当性確認
独立した立場で確認 → 監査
使いやすさを高める → 使用性向上
```

共通フレーム全体の用語を暗記するより、このように選択肢を切る基準を持つ方が実戦的です。

## 科目Bでどう使う？

科目Bで共通フレームのプロセス名だけを直接問う場面は多くありません。

ただし、システム開発の長い説明文を読むときに、**今どの段階の話をしているか** を整理する力は役立ちます。

例えば、次のように読み分けます。

```text
要求を整理している → 要件定義寄り
設計・実装している → 開発寄り
稼働後に対応している → 運用・保守寄り
正しさを確認している → 支援プロセス寄り
進捗やリスクを見ている → マネジメント寄り
```

科目Bでは、問題文の登場人物や工程が多くなることがあります。

そのときに、共通フレームの考え方を使って、**これは開発の話か、運用の話か、確認の話か、管理の話か** を分けると読みやすくなります。

## よくある誤解・混同

共通フレームでよくある誤解は、**開発手法の一つだと思ってしまうこと** です。

| 誤解 | 正しい理解 |
|---|---|
| 共通フレームはウォーターフォールやアジャイルの一種 | 開発手法ではなく、作業を共通のプロセスとして整理する枠組み |
| 共通フレームは開発者だけが使う | 発注者、利用者、開発者、運用担当など関係者全体に関わる |
| 支援プロセスはおまけなので出題されにくい | 検証、妥当性確認、監査などは選択肢で問われやすい |
| 検証と妥当性確認は同じ | 検証は仕様どおり、妥当性確認は利用者の目的どおり |
| 監査は開発作業の一部 | 監査は独立した立場で適合性を見る活動 |

特に、共通フレームの問題では、用語が少し堅く見えます。

でも、試験で見るべきポイントはシンプルです。

```text
契約の話？
開発の話？
運用・保守の話？
管理の話？
正しさを確認する話？
```

このように分けると、選択肢を落ち着いて読めます。

## まとめ（試験直前用）

- 共通フレームは、システム開発や運用の作業を共通のプロセスで整理する枠組み
- 開発手法そのものではなく、関係者が同じ言葉で作業を整理するためのもの
- 合意、企画、開発、運用、保守、マネジメント、支援などの視点がある
- 支援プロセスでは、検証、妥当性確認、監査、使用性向上を切り分ける
- 「仕様どおりか」と「利用者の目的どおりか」を分けると得点しやすい
