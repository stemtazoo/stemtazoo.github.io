## まず結論

コミットとは、**トランザクションの処理結果を確定する操作**です。

ロールバックとは、**途中で失敗した処理を取り消し、処理前の状態に戻す操作**です。

基本情報技術者試験では、次のように切り分けると判断しやすくなります。

| 用語 | 意味 | 判断の合図 |
|---|---|---|
| コミット | 処理結果を確定する | 正常終了、確定、反映 |
| ロールバック | 処理を取り消して戻す | 異常終了、取消、処理前へ戻す |

特に、**全部成功したらコミット、途中で失敗したらロールバック** と考えると分かりやすいです。

## 直感的な説明

コミットとロールバックは、ネットショップの注文処理で考えると分かりやすいです。

例えば、商品を注文するとき、データベースでは次のような処理が行われます。

```text
1. 在庫数を減らす
2. 注文履歴を追加する
3. 請求情報を登録する
```

この3つがすべて成功したら、注文は成立します。

```text
全部成功
  → コミット
  → 注文結果を確定
```

これが **コミット** です。

一方で、在庫数は減らせたけれど、請求情報の登録で失敗したとします。

このとき、在庫だけ減った状態を残すと困ります。

```text
在庫数だけ減る
注文履歴が中途半端
請求情報がない
```

このような中途半端な状態を残さないために、処理前の状態へ戻します。

```text
途中で失敗
  → ロールバック
  → 処理前の状態へ戻す
```

これが **ロールバック** です。

イメージとしては、

```text
コミット　　：これで確定する
ロールバック：なかったことに戻す
```

です。

## 定義・仕組み

トランザクションとは、データベースに対する一連の処理を、ひとまとまりの単位として扱う考え方です。

例えば、銀行振込では、出金と入金をまとめて1つの処理として扱います。

```text
トランザクション開始
  Aさんの残高を減らす
  Bさんの残高を増やす
コミット
```

このように、すべての処理が正常に終わったら、最後にコミットして結果を確定します。

途中で問題が起きた場合は、ロールバックして処理前の状態に戻します。

```text
トランザクション開始
  Aさんの残高を減らす
  Bさんの残高を増やす途中で失敗
ロールバック
```

この仕組みによって、データベースに中途半端な状態が残りにくくなります。

| 操作 | 何をする？ | 関係が深いACID特性 |
|---|---|---|
| コミット | 処理結果を確定する | 耐久性 |
| ロールバック | 処理前の状態に戻す | 原子性 |

コミットされた結果は、基本的に確定済みの結果として扱われます。

そのため、コミットは **耐久性** と関係が深いです。

ロールバックは、途中まで実行された処理を取り消すため、**原子性** と関係が深いです。

このテーマは、基本情報技術者試験の「データベース」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

## 科目Aでどう出る？

科目Aでは、コミットとロールバックの意味を問う問題が出やすいです。

次の表で切り分けると判断しやすくなります。

| 問題文の表現 | 選ぶ用語 |
|---|---|
| トランザクションの処理結果を確定する | コミット |
| 更新結果をデータベースへ反映する | コミット |
| 処理を取り消して処理前の状態へ戻す | ロールバック |
| 障害発生時に未確定の処理を取り消す | ロールバック |
| すべて成功なら確定、失敗なら取消 | トランザクション処理 |

特に注意したいのは、**ロールバックは「前へ進める処理」ではなく「戻す処理」** という点です。

名前に `back` が入っているので、戻すイメージで覚えるとよいです。

```text
commit   ：確定する
rollback ：戻す
```

## 科目Bでどう使う？

科目Bでは、コミットとロールバックを、処理の流れの中で読むことがあります。

見るポイントは、次の3つです。

```text
1. どこからトランザクションが始まるか
2. どこでコミットされるか
3. 失敗時にどこまで戻るか
```

例えば、次の処理を考えます。

```text
トランザクション開始
  在庫数を減らす
  注文履歴を追加する
  請求情報を登録する
コミット
```

この場合、コミットまで到達すれば、在庫数、注文履歴、請求情報の更新が確定します。

一方で、請求情報の登録で失敗した場合は、ロールバックして、在庫数や注文履歴の更新も取り消します。

```text
在庫数を減らす
注文履歴を追加する
請求情報で失敗
  → ロールバック
  → 在庫数も注文履歴も処理前へ戻す
```

科目Bでは、頭の中だけで追うと混乱しやすいです。

次のように、確定前か確定後かを分けて読むと安全です。

| 状態 | 読み方 |
|---|---|
| コミット前 | まだ確定していない |
| コミット後 | 結果が確定した |
| ロールバック後 | トランザクション開始前に戻った |

## よくある誤解・混同

### 誤解1：コミット前の更新も確定済みだと思ってしまう

トランザクション中の更新は、コミットされるまでは確定とは考えません。

処理の途中でエラーが起きると、ロールバックで取り消されることがあります。

```text
更新した
でも、まだコミットしていない
  → 確定ではない
```

試験では、**コミットされたかどうか** を確認します。

### 誤解2：ロールバックを「最後の1処理だけ取り消す」と考える

ロールバックは、基本的にトランザクション内の処理を、処理前の状態へ戻す操作です。

最後の1行だけを取り消す、というより、対象のトランザクション全体を戻すイメージです。

```text
トランザクション開始
  処理A
  処理B
  処理Cで失敗
ロールバック
  → 処理A、処理Bも含めて取り消す
```

この考え方は、ACID特性の **原子性** とつながります。

### 誤解3：コミットとロールフォワードを混同する

コミットは、トランザクションの結果を確定する操作です。

ロールフォワードは、障害復旧でログなどを使って、ある時点以降の更新を再実行する考え方です。

| 用語 | ざっくりした意味 |
|---|---|
| コミット | 処理結果を確定する |
| ロールバック | 処理前へ戻す |
| ロールフォワード | ログを使って更新を進めて復旧する |

ロールバックとロールフォワードは名前が似ていますが、向きが違います。

```text
ロールバック　　：戻す
ロールフォワード：進める
```

### 誤解4：ロールバックすれば必ずすべての障害に対応できると思う

ロールバックは、未確定の処理を取り消すための考え方です。

障害復旧全体では、バックアップ、ログ、ロールフォワードなども関係します。

FE試験では、問題文が何を聞いているかを見ます。

```text
未確定処理を取り消す → ロールバック
確定済みの更新を復旧する → ロールフォワード
```

と切り分けます。

## まとめ（試験直前用）

- コミットは、トランザクションの結果を確定する操作
- ロールバックは、失敗時に処理前の状態へ戻す操作
- コミット前の更新は、まだ確定とは限らない
- ロールバックは、原子性と関係が深い
- ロールフォワードは「進める復旧」なので、ロールバックと混同しない
