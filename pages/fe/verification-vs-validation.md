## まず結論

検証と妥当性確認は、どちらも「正しいかを確認する」活動ですが、**見る基準が違います**。

| 用語 | 判断の軸 |
|---|---|
| 検証 | 仕様・要件・設計どおりに作れているか |
| 妥当性確認 | 利用者の目的・使用方法に合っているか |

基本情報技術者試験では、**仕様どおりなら検証、利用者の意図どおりなら妥当性確認** と考えると選択肢を切りやすいです。

## 直感的な説明

検証と妥当性確認は、家を建てる場面で考えると分かりやすいです。

設計図に「窓はこの位置、ドアはこの大きさ」と書かれているとします。

その設計図どおりに作れているかを確認するのが、検証です。

一方で、その家が「住む人にとって本当に使いやすいか」「生活の目的に合っているか」を確認するのが、妥当性確認です。

```text
検証：設計図どおりに作ったか
妥当性確認：住む人が求めていた家になっているか
```

英語の定番フレーズでは、次のように整理できます。

```text
Verification: Are we building the product right?
Validation: Are we building the right product?
```

日本語では、**正しく作っているか** が検証、**正しいものを作っているか** が妥当性確認です。

## 定義・仕組み

検証は、成果物やサービスが、規定された要求、仕様、設計、計画などを満たしているかを確認する活動です。

妥当性確認は、成果物やサービスが、指定された使用方法や利用者の目的に対して適切かを確認する活動です。

| 観点 | 検証 | 妥当性確認 |
|---|---|---|
| 見る基準 | 要求、仕様、設計、規定 | 利用者の目的、使用方法、意図 |
| 見る方向 | 作ったものが決まりに合っているか | 作ったものが使う人に合っているか |
| 代表的な表現 | 規定要求を満たす | 利用者の視点で確認する |
| 英語 | Verification | Validation |
| 覚え方 | 仕様どおり？ | 目的どおり？ |

例えば、ログイン画面を作る場合を考えます。

| 確認内容 | どちら？ |
|---|---|
| パスワード欄が仕様書どおりに8文字以上を受け付けるか | 検証 |
| 利用者が迷わずログインできるか | 妥当性確認 |
| 設計書に書かれたエラーメッセージが表示されるか | 検証 |
| 実際の利用場面でエラー内容が分かりやすいか | 妥当性確認 |

このテーマは、基本情報技術者試験の「ソフトウェア開発」や「システム開発技術」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

## 科目Aでどう出る？

科目Aでは、検証プロセスと妥当性確認プロセスの説明を選ばせる問題で出題されやすいです。

判断するときは、問題文の中にある **基準になる言葉** を探します。

| 問題文・選択肢の表現 | 判断 |
|---|---|
| 規定要求を満たしているか | 検証 |
| 仕様を適切に反映しているか | 検証 |
| 設計どおりに作られているか | 検証 |
| 利用者の視点から意図したものか | 妥当性確認 |
| 指定された使用方法に対する要件を満たすか | 妥当性確認 |
| 利用者の目的に合っているか | 妥当性確認 |

試験では、次の一言で切ると楽です。

```text
仕様・要件・設計 → 検証
利用者・使用方法・目的 → 妥当性確認
```

どちらも「確認する」なので、確認という言葉だけでは判断できません。

**何を基準に確認するのか** を見るのが大切です。

## 科目Bでどう使う？

科目Bでは、長い問題文の中で、確認作業の目的を読み取る場面があります。

例えば、次のように読むと切り分けやすいです。

```text
設計書に書かれた処理と一致しているかを確認した
→ 検証

利用者が実際の業務で目的を達成できるかを確認した
→ 妥当性確認
```

また、テスト工程の文章でも役立ちます。

```text
仕様に対して正しいか
→ 検証寄り

利用者が使う場面で正しいか
→ 妥当性確認寄り
```

科目Bでは、用語名を直接聞かれなくても、**開発側の基準で見ているのか、利用者側の基準で見ているのか** を意識すると読みやすくなります。

## よくある誤解・混同

検証と妥当性確認で一番多い誤解は、どちらも「正しいかを確認する」ので同じだと思ってしまうことです。

| 誤解 | 正しい理解 |
|---|---|
| 検証も妥当性確認も同じ | 基準が違う |
| 利用者の視点でも仕様書に合えば検証 | 利用者の目的を見るなら妥当性確認 |
| 仕様どおりなら必ず利用者にとって正しい | 仕様自体が利用目的からずれていることもある |
| 妥当性確認は単なる最終確認 | 利用者の使用方法や目的に合うかを見る確認 |
| 検証はテストだけを指す | レビューや確認など広い活動で使われる |

特に、次の2つはセットで覚えると強いです。

```text
検証：正しく作っているか
妥当性確認：正しいものを作っているか
```

また、監査プロセスとも混同しやすいです。

| 用語 | 見るポイント |
|---|---|
| 検証 | 仕様や要件に合うか |
| 妥当性確認 | 利用者の目的に合うか |
| 監査 | 独立した立場で適合性を見る |

「独立した立場」とあれば、検証や妥当性確認ではなく、監査を疑います。

## まとめ（試験直前用）

- 検証は、仕様・要件・設計どおりに作れているかを見る
- 妥当性確認は、利用者の目的や使用方法に合っているかを見る
- 「仕様どおり？」なら検証
- 「利用者の意図どおり？」なら妥当性確認
- どちらも確認なので、確認対象の基準を読む
- 「独立した立場」なら監査プロセスを疑う
