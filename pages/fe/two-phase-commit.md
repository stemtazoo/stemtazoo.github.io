## まず結論

2相コミットとは、**分散データベースで、複数サイトの更新を「全員で成功」または「全員で中止」にそろえる仕組み**です。

基本情報技術者試験では、2相コミットは **分散データベースのトランザクションを一貫して確定する方式** として出題されやすいです。

ポイントは、いきなり更新を確定しないことです。

まず、すべてのサイトに「更新してよいか」を確認します。

そのあと、全員が更新可能ならコミットし、1つでも無理ならロールバックします。

## 直感的な説明

2相コミットは、**みんなで同時に提出する前に、全員へ確認を取る仕組み** と考えると分かりやすいです。

例えば、複数の支店で同時に在庫データを更新するとします。

ある支店だけ更新に成功し、別の支店では失敗すると、データのつじつまが合わなくなります。

そこで、まず全支店に聞きます。

```text
調停者：
「みんな、更新できる？」
```

全員が「できます」と答えたら、次に確定します。

```text
調停者：
「では、全員コミットして」
```

もし1つでも「できません」と答えたら、全員で中止します。

```text
調停者：
「では、全員ロールバックして」
```

このように、2相コミットは **確認してから確定する** 方式です。

## 定義・仕組み

2相コミットは、分散データベース環境で、複数のサイトにまたがるトランザクションの原子性と一貫性を保つための方式です。

2相コミットでは、中心となる **調停者** と、更新に参加する **参加者** が登場します。

| 役割 | 意味 |
|---|---|
| 調停者 | コミットするか中止するかを全体で決める役割 |
| 参加者 | 実際にデータを更新する各サイトやノード |
| 第1相 | 更新できるかを確認する段階 |
| 第2相 | コミットまたはロールバックを実行する段階 |

2相コミットは、名前のとおり2つの段階で進みます。

### 第1相：準備・確認

調停者が、各参加者に更新可能かどうかを問い合わせます。

参加者は、自分のサイトで更新できる状態かを確認し、可能なら「準備完了」と返します。

この段階では、まだ最終確定しません。

### 第2相：確定・中止

すべての参加者が準備完了なら、調停者は全参加者にコミットを指示します。

1つでも準備できない参加者がいれば、調停者は全参加者にロールバックを指示します。

このテーマは、基本情報技術者試験の「データベース」や「トランザクション処理」と関係する内容です。公式の出題範囲やシラバスは、[IPA：基本情報技術者試験](https://www.ipa.go.jp/shiken/kubun/fe.html) から確認できます。

## 科目Aでどう出る？

科目Aでは、2相コミットは、分散データベースの更新方式として出題されやすいです。

判断するときは、**複数サイトに更新可能かを問い合わせ、全員の確認後に更新するか** を見ます。

| 問題文の表現 | 選ぶ用語 |
|---|---|
| 複数サイトに更新可能か問い合わせる | 2相コミット |
| すべてのサイトが更新可能であることを確認してから更新する | 2相コミット |
| 分散DBでトランザクションの整合性を保つ | 2相コミット |
| 他のトランザクションを待たせる | 排他制御 |
| 更新前ログで開始直前に戻す | ロールバック |
| 更新後ログで障害直前まで復旧する | ロールフォワード |

例えば、次のような説明は2相コミットです。

```text
一連のトランザクション処理を行う複数サイトに更新可能かどうかを問い合わせ，
すべてのサイトが更新可能であることを確認した後，
データベースの更新処理を行う。
```

この説明では、**複数サイト**、**更新可能か問い合わせる**、**すべて確認した後に更新** が決め手です。

科目Aでは、**分散DBで全員に確認してから確定するなら2相コミット** と考えると選択肢を切りやすいです。

## 科目Bでどう使う？

2相コミットは、科目Bの擬似言語トレース問題で中心になることは多くありません。

ただし、トランザクション処理や障害対応の説明を読むときに、2相コミットの流れを知っていると整理しやすいです。

例えば、3つのサイト A、B、C が同じトランザクションに参加するとします。

```text
第1相：
Aに確認 → 準備OK
Bに確認 → 準備OK
Cに確認 → 準備OK

第2相：
Aにコミット指示
Bにコミット指示
Cにコミット指示
```

この場合、全員が準備OKなので、全員でコミットします。

一方で、1つでも失敗した場合は次のようになります。

```text
第1相：
Aに確認 → 準備OK
Bに確認 → 準備NG
Cに確認 → 準備OK

第2相：
Aにロールバック指示
Bにロールバック指示
Cにロールバック指示
```

2相コミットでは、あるサイトだけコミットし、別のサイトだけ中止する状態を避けようとします。

科目Bや長文問題で読むときは、次の順で整理すると分かりやすいです。

1. 複数のサイトやノードが関係しているかを見る
2. 1つのトランザクションが複数サイトにまたがるかを見る
3. まず更新可能か確認しているかを見る
4. 全員OKならコミット、1つでもNGならロールバックかを見る

2相コミットは、**分散環境でトランザクションを中途半端に終わらせないための仕組み** と考えると理解しやすいです。

## よくある誤解・混同

2相コミットで混同しやすいのは、排他制御、ロールバック、ロールフォワードです。

| 用語 | 意味 | 判断の合図 |
|---|---|---|
| 2相コミット | 分散DBで全参加者の確認後にコミットする方式 | 複数サイト、更新可能か確認、全員で確定 |
| 排他制御 | 同時更新による不整合を防ぐ制御 | ロック、待機、他のトランザクションを止める |
| ロールバック | 処理を取り消して開始前の状態へ戻す | 更新前ログ、取り消し、元に戻す |
| ロールフォワード | バックアップ後の更新を再実行して復旧する | 更新後ログ、障害直前まで復旧 |
| コミット | トランザクションの更新を確定する | 確定、反映 |

排他制御は、同時に同じデータを更新して不整合が起きないようにする仕組みです。

2相コミットは、分散環境で複数サイトの更新をそろえる仕組みです。

また、ロールバックとロールフォワードも混同しやすいです。

- ロールバック：処理を取り消して前に戻す
- ロールフォワード：バックアップから復元した後、ログを使って障害直前まで進める

2相コミットは、ロールバックを使うことがあります。  
ただし、2相コミットそのものは「分散環境で全員の確定をそろえる手順」です。

## さらに注意したいひっかけ

2相コミットでは、次のようなひっかけにも注意します。

| ひっかけ表現 | なぜ注意する？ |
|---|---|
| 1つのタスクが終わるまで他を待たせる | 排他制御の説明 |
| 更新前ログで開始直前に戻す | ロールバックの説明 |
| 更新後ログで障害直前まで復旧する | ロールフォワードの説明 |
| すぐに全サイトへ更新を確定する | 2相コミットは先に準備確認を行う |
| 一部のサイトだけコミットする | 2相コミットでは全員で確定または全員で中止を目指す |
| 単一DB内の通常更新だけを指す | 2相コミットは分散DBで出やすい |

特に、**2相コミットはロールバックそのものではない** という点は重要です。

2相コミットでは、全員が準備できなければロールバックを指示します。

しかし、問題文が「更新前ログを使って元に戻す」と言っているなら、ロールバックを選びます。

一方で、問題文が「複数サイトに更新可能か問い合わせる」と言っているなら、2相コミットを疑います。

## まとめ（試験直前用）

- 2相コミットは、分散DBで複数サイトの更新をそろえる仕組み
- 第1相で更新可能か確認し、第2相でコミットまたはロールバックする
- すべての参加者がOKならコミットする
- 1つでもNGなら全体をロールバックする
- 排他制御は、同時更新を防ぐために待たせる仕組み
- ロールバックは、更新前ログで元に戻す処理
- ロールフォワードは、更新後ログで障害直前まで進める復旧処理
